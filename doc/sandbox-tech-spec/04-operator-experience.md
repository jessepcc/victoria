## 1. Purpose and Scope

This spec covers the messaging gateway and operator-facing UX layer: how review packets reach the operator, how operator replies are parsed and structured, and how the resulting Correction is handed back to the execution plane. It defines the firm boundary between the gateway and the execution plane, and the three inter-component contracts the rest of the system depends on.

### Firm boundary

| Owned by this spec | Owned by peers |
|---|---|
| Inbound and outbound message routing (WhatsApp + Telegram) | Temporal workflow internals (Execution Architect) |
| Review packet rendering and delivery (the `SendApprovalPacket` activity hands off to the gateway) | `SendApprovalPacket` activity orchestration (Execution Architect) |
| Inbound classification: button enum, scope hint extraction, follow-up flow, dead-letter routing | Semantic parsing of `free_text` + `agent_input` → `parsed_conditions` + `parsed_action` (Learning Architect — RESOLVED-Parser-split) |
| Emitting `Correction` payload conforming to learning §3 | Persisting `Correction` row, candidate matching, confidence scoring (Learning Architect) |
| Translating `Correction` into Temporal `CaseApproved` / `CaseRejected` signal payloads | Temporal signal handlers and signal dedup invariants (Execution Architect) |
| Per-tenant number routing table (`channel_bindings`) | Tenant lifecycle, JWT issuance, operator OTP verification (Control Plane Architect) |
| Multi-turn follow-up state (gateway-scoped, TTL ≤ 1 hour) | Long-running workflow state beyond expiry (Execution Architect) |
| Mobile artifact preview pages: signed-JWT minting using the gateway's preview signing key | Artifact generation pipeline, object-store path layout (Execution Architect) |
| Outbound delivery idempotency, outbound queue during session outage | Activity-level idempotency keys (Execution Architect) |
| Audit events: packet sent, correction received, expiry, dead-letter, session disconnect, security_violation_inbound (gateway writes these to per-tenant `audit_events` table) | Audit table schema, immutability triggers, retention policy (Control Plane / Learning Architect — shared infra; gateway is one writer among several) |
| WhatsApp session: QR pairing, whatsmeow store, reconnect, alerting | Ctrl-plane provisions session storage; gateway operates whatsmeow in-process (§4.2, §5.7) |

**Assumption A1** (needs Execution Architect confirmation): `SendApprovalPacket` activity (per exec-plane §3.4) hands off the approval packet to the gateway via a defined transport. The activity completes once the gateway confirms enqueue; redelivery is idempotent on `(case_run_id, decision_point_id)`.

**Assumption A2** (needs Control Plane Architect confirmation): The Control Plane's `TenantProvisioningWorkflow` (per ctrl-plane §5.2) extends to provision per-tenant WhatsApp session storage and call the gateway's `POST /channel-bindings` endpoint. Gateway owns the `channel_bindings` row and the in-process whatsmeow session; control plane owns the tenant lifecycle.

**Assumption A3** (CONFIRMED by learning §3): Gateway emits a `Correction` payload exactly matching the learning-loop's `Correction` JSON schema. Gateway does not produce RuleCandidates and does not parse `parsed_conditions` or `parsed_action`. See §5 for the aligned shape.

**Assumption A4** (needs Control Plane Architect confirmation): Operator OTP verification is performed by ctrl-plane's `POST /auth/operator/verify-otp`. The gateway is a *caller*, not an issuer, of operator JWTs. The gateway holds (a) WhatsApp session keys, (b) Temporal client credentials for signal-send, (c) preview-JWT signing keys — none of which are operator auth tokens.

## 2. Contract Surfaces (read first)

Three contracts that all other components depend on. Described here as canonical JSON sketches; the prose sections below elaborate on each.

### 2.1 Outbound: ReviewPacket (R3 — `mode` field added per RESOLVED-N1)

```json
{
  "packet_id": "pkt_7a3c",
  "schema_version": "1.1",
  "tenant_id": "t_123",
  "case_run_id": "cr_456",
  "decision_point_id": "dp_789",
  "workflow_type": "quote_drafting",
  "mode": "sandbox",
  "trigger": {
    "summary": "<human-readable summary of what triggered this>",
    "source_channel": "email",
    "received_at": "2026-04-27T08:00:00Z"
  },
  "facts": [
    { "label": "<fact label>", "value": "<extracted value>", "confidence": 0.91 }
  ],
  "planned_action": {
    "type": "send_quote",
    "description": "<human-readable description of intended action>",
    "is_destructive": false,
    "requires_approval": true
  },
  "artifact_preview": {
    "type": "pdf",
    "preview_url": "https://preview.victoria.app/a/pkt_7a3c?token=<signed-jwt>",
    "inline_thumbnail_url": null,
    "expires_at": "2026-04-27T10:00:00Z"
  },
  "button_set": ["approve", "wrong_facts", "wrong_action", "missing_condition", "use_different_template", "add_note"],
  "expires_at": "2026-04-28T08:00:00Z",
  "idempotency_key": "pkt_7a3c:deliver:1"
}
```

**RESOLVED-N1.** `mode: "sandbox" | "live"` is the canonical encoding across all four specs. Default-deny: a missing or unknown value is rendered as sandbox (gateway never falls open to live). Set by the execution plane when it calls the gateway's outbound endpoint (per exec-plane INV-T5).

**Operator UX consequence:** when `mode == "sandbox"` the channel adapter may visually flag the message (e.g., a "[SANDBOX]" tag in the rendered button group or a leading line in the message body). The exact rendering is an adapter-level concern; the contract is that sandbox runs are not visually indistinguishable from live runs.

`button_set` carries the enum values, not display strings. The R1 `metadata.run_mode` field is removed (superseded by `mode`).

### 2.2 Inbound + Signal: ApprovalSignalEnvelope (RESOLVED-C13)

**The gateway emits exactly one wire object per resolved operator reply: a Temporal signal whose payload is the full Correction.** This is the single canonical envelope. Exec-plane's `PersistCorrection` activity (the sole writer of the `corrections` row per learning §3.5 and RESOLVED-C10) reads this payload and writes the row without any additional fetch.

**Signal name mapping (matches exec-plane §7.3):**

| `action_button` | Temporal signal name |
|---|---|
| `approve` | `CaseApproved` |
| `wrong_facts`, `wrong_action`, `missing_condition`, `use_different_template`, `add_note` | `CaseRejected` |

**`ApprovalSignalEnvelope` (canonical 16-field payload — R5 final, ratified by exec-plane R4 §7.4 verbatim):**

```json
{
  "schema_version": "1",
  "signal_id": "<sha256 per ctrl-plane §17.2 components: tenant_id, packet_id, action_button, raw_inbound_message_id>",
  "idempotency_key": "<sha256 per ctrl-plane §17.2 components: tenant_id, decision_point_id, packet_id, action_button>",
  "packet_id": "pkt_7a3c",
  "case_run_id": "cr_456",
  "decision_point_id": "dp_789",
  "tenant_id": "t_123",
  "operator_id": "op_whatsapp:+61400000000",
  "channel": "whatsapp",
  "source_message_id": "wa_5G3K7Z1ABCDEF",
  "raw_inbound_message_id": "wa:wa_5G3K7Z1ABCDEF",
  "ts": "2026-04-27T09:14:00Z",
  "action_button": "wrong_action",
  "free_text": "Should have held and asked for more photos.",
  "follow_up_answer": "always when client is new and photos are incomplete",
  "scope_hint": "tenant",
  "parser_method": "button",
  "parser_confidence": 1.0
}
```

(Sixteen top-level wire fields. Exec-plane R4 §7.4 ratifies this shape verbatim: "Operator-ux delivered the 16-field envelope ... no additional fields are required.")

**Two fields handled out-of-band (NOT on the wire):**
- **`correction_id`:** generated by `PersistCorrection` when it inserts the row. The activity-level idempotency key `{case_run_id}:{correction_id}` (exec §3.4) is exec-plane internal. R4 had this on the wire; reverted in R5 per exec-plane §7.4 final.
- **`parse_status`:** gateway-internal precondition. Only envelopes with gateway-internal `parse_status: resolved` are emitted as Temporal signals. `needs_followup` and `dead_lettered` are gateway-side terminal states (I-07); the receiver never sees a non-resolved envelope. R4 had this on the wire; reverted in R5 per exec-plane §7.4 final.

**Field-naming locks (R5 — closes OPEN-Envelope-Naming).** Exec-plane R4 §7.4 ratified the gateway's R3 names verbatim. Four cosmetic questions resolved:

| Lock | Rationale |
|---|---|
| `ts` (not `received_at`) | Concise; ratified by exec-plane R4 §7.4. |
| `action_button` (not `button_action`) | Matches learning's `corrections.action_button` column (learning §3.3). No rename at the boundary. |
| `parser_method` (not `parse_method`) | Per exec-plane R4 RESOLVED-C13-FOLLOWUP: gateway emits the *parser stage* name (`parser_method`); `corrections.parse_method` documents the *parsed result*. PersistCorrection renames at the deserialize boundary — deliberate, documented, not a fork. |
| `parser_confidence` (not `parse_confidence`) | Same rationale. |

**`scope_hint` lock (R5 — closes OPEN-Envelope-ScopeHint).** Gateway emits **canonicalized values** (`case | tenant | null`), not the raw `always | this_case | null` form. The pre-mapping happens in the gateway's follow-up flow (§7.2): operator's "always when …" → `tenant`; "this case" → `case`. PersistCorrection writes the value verbatim to `corrections.scope_hint` (no remap; matches learning §3.3 column constraint `scope_hint IN ('case','tenant',NULL)`). Exec-plane R4 §7.4 row 16 confirms: "Operator-ux already uses learning-loop's mapping ... no remap in `PersistCorrection`."

**`condition_hints` not on the wire.** RESOLVED-Parser-split (R2 binding) keeps Stage B with learning. Gateway omits the field; learning's Stage B parser builds `parsed_conditions` from `decision_point.agent_input` post-row-write.

**Field semantics:**

| Field | Type / Enum | Source | Used by |
|---|---|---|---|
| `schema_version` | string `"1"` | gateway | all |
| `signal_id` | sha256 per ctrl §17.2 | gateway-derived at parse time | Temporal signal-handler dedup (devils-advocate INV-06) |
| `idempotency_key` | sha256 per ctrl §17.2 | gateway-derived | learning `corrections.idempotency_key` UNIQUE |
| `packet_id` | string | exec-plane (originating ReviewPacket) | reconciliation with outbound packet record |
| `case_run_id`, `decision_point_id`, `tenant_id` | strings | gateway (from `channel_bindings` + packet ref) | workflow routing, RLS scope |
| `operator_id` | `op_whatsapp:<E.164>` \| `op_telegram:<chat_id>` | gateway | learning `corrections.operator_id`, audit `actor_id` |
| `channel` | `"whatsapp" \| "telegram"` | gateway | audit `payload.channel` |
| `source_message_id` | provider message ID | gateway | secondary dedup, audit `payload.source_message_id` |
| `raw_inbound_message_id` | `<provider>:<msg_id>` | gateway | learning `corrections.raw_inbound_message_id` per §3.3 |
| `ts` | ISO 8601 UTC | gateway | learning `corrections.created_at` |
| `action_button` | enum (6 values; see below) | gateway parser Stage A | drives signal name + `corrections.action_button` |
| `free_text` | string \| null | operator verbatim | learning `corrections.free_text` |
| `follow_up_answer` | string \| null | operator verbatim (newline-concatenated if multi-turn) | learning `corrections.follow_up_answer` |
| `scope_hint` | `"case" \| "tenant" \| null` (canonicalized) | gateway parser Stage A | learning `corrections.scope_hint` |
| `parser_method` | `"button" \| "button_fallback" \| "text_match"` | gateway | learning `corrections.parse_method` (renamed at PersistCorrection boundary) |
| `parser_confidence` | float `[0.0, 1.0]` | gateway | learning `corrections.parse_confidence` (renamed at PersistCorrection boundary) |

**`action_button` enum (canonical, stable across all specs):** `approve`, `wrong_facts`, `wrong_action`, `missing_condition`, `use_different_template`, `add_note`.

**One envelope, two consumers, no extra fetch:**

```
gateway resolves Correction (parse_status: resolved)
    │
    └──> Temporal SignalWorkflow(workflow_id=case_run_id, signal=CaseApproved|CaseRejected, payload=envelope)
              │
              ├──> exec-plane signal handler dedups on signal_id (devils-advocate INV-06)
              │       └──> PersistCorrection activity reads envelope, writes corrections row
              │             - corrections.id ← server-generated UUID
              │             - corrections.idempotency_key ← envelope.idempotency_key
              │             - corrections.action_button ← envelope.action_button
              │             - corrections.free_text ← envelope.free_text
              │             - corrections.follow_up_answer ← envelope.follow_up_answer
              │             - corrections.scope_hint ← envelope.scope_hint (mapping per learning §3.3)
              │             - corrections.parse_method ← envelope.parser_method
              │             - corrections.parse_confidence ← envelope.parser_confidence
              │             - corrections.raw_inbound_message_id ← envelope.raw_inbound_message_id
              │             - corrections.operator_id ← envelope.operator_id
              │             - corrections.created_at ← envelope.ts
              │             - all FK columns ← envelope.{case_run_id, decision_point_id, tenant_id}
              │
              └──> learning Stage B parser reads corrections row (NOT the envelope) for canonicalization
```

The gateway does NOT separately POST a Correction to learning — exec-plane's PersistCorrection is the sole writer of the row.

**Idempotency layering (one envelope, two dedup keys):**

| Key | Layer | Dedups against |
|---|---|---|
| `signal_id` | Temporal signal handler (workflow-side, INV-A2) | duplicate signal delivery from gateway retries or Temporal redelivery |
| `idempotency_key` | learning `corrections.idempotency_key` UNIQUE | duplicate `corrections` row even if signal was delivered to a freshly restarted workflow |

Both must hold; either alone is insufficient.

**Cross-spec consistency (R5 confirmed):** exec-plane R4 §7.4 consumes this 16-field shape verbatim; learning §3.3 column-mapping receives the row from PersistCorrection. Learning §3.1's prose still references the legacy R2 `StructuredCorrection` JSON; learning's column-mapping table in §3.3 is the authoritative consumer surface.

### 2.3 Why direct-to-Temporal (RESOLVED-C3, ratified)

- Exec-plane has no inbound HTTP surface (exec-plane §1).
- Ctrl-plane outage must not block operator approvals (ctrl-plane §6.6).
- Auditability is preserved via the audit-emit path (§9.6 here, ctrl-plane §6.6).

## 3. Test Strategy (TDD) — Invariants and Contract Tests

### Invariants

Every invariant must have a corresponding contract test before any implementation is merged.

| # | Invariant |
|---|---|
| I-01 | Every inbound WhatsApp or Telegram message produces exactly one of: `{ack_only, follow_up_question_emitted, structured_correction_emitted, dead_lettered}`. No silent drops. |
| I-02 | Duplicate webhook delivery for the same provider `message_id` within the dedup window is a no-op; the gateway returns 200 and emits nothing. |
| I-03 | No `packet_id` results in more than one Temporal signal delivery for the same `decision_point_id`. |
| I-04 | A reply received after `packet.expires_at` produces a `stale_reply` audit event; it does not deliver a Temporal signal. |
| I-05 | Outbound delivery for a given `(tenant_id, packet_id)` is at-least-once with a provider-side dedup key; the gateway does not double-deliver to the same provider thread within a dedup window. |
| I-06 | The channel adapter interface is identical for WhatsApp and Telegram; the same contract test suite runs against both adapters. |
| I-07 | A Correction with gateway-internal `parse_status: dead_lettered` never produces a Temporal signal and is not written to the learning loop's `corrections` table. |
| I-08 | Every outbound packet and every inbound correction is written to the audit log before any downstream action is taken. |
| I-09 | Signed preview URLs contain the `tenant_id` in the JWT claims and reject requests where the signed `tenant_id` does not match the serving tenant. |
| I-10 | Follow-up state for `(tenant_id, packet_id)` is cleared on: correction resolved, correction dead-lettered, or TTL expiry — whichever comes first. |
| I-11 | No message loss across a WhatsApp session disconnect of duration ≤ 30 minutes: outbound packets queued during the outage deliver exactly once on reconnect; inbound replies received via webhook redelivery dedup correctly. |
| I-12 | Inbound webhook `tenant_id` is derived solely from the `provider_number` lookup in `channel_bindings`. Any `tenant_id` field present in the webhook payload is ignored; mismatches between message content and the bound tenant are dropped with a `security_violation_inbound` audit event. |
| I-13 | Gateway's emitted `ApprovalSignalEnvelope` validates against the 16-field schema in §2.2; missing or extra fields fail CI. |
| I-14 | The gateway never populates `parsed_conditions` or `parsed_action`; learning's Stage B owns canonicalization. |
| I-15 (R3) | A single resolved operator reply produces exactly ONE outbound Temporal signal call. The gateway does NOT additionally POST to learning's correction ingest endpoint. PersistCorrection is the sole writer. |
| I-16 (R3) | The gateway loads the Temporal signal-only credential keyed by inbound `tenant_id`, not by a gateway-process-global credential. A burst of N signals across N tenants invokes N distinct credential loads (cached with TTL per §10.4). |
| I-17 (R3) | Gateway audit-event writes go through ctrl-plane's `POST /internal/audit/events`; the gateway does not hold per-tenant DB credentials and does not write directly to the `audit_events` table. |
| I-18 (R3) | When ReviewPacket `mode` is missing or unrecognized, the rendered packet treats the run as sandbox (default-deny). The mode value reflected in operator UX never silently falls open to "live". |
| I-19 (R3) | The gateway-held Temporal credential for tenant T cannot start, terminate, reset, or describe-workflow on any task queue; it can only signal an existing workflow. |

### Contract tests

- **CT-Outbound-01:** Given a valid `ReviewPacket`, the channel adapter produces a correctly formatted outbound message for WhatsApp and for Telegram; assertions on field mapping, not message text.
- **CT-Outbound-02:** Given a `ReviewPacket` with `artifact_preview.type = pdf`, the signed URL in the delivered message contains a JWT with `exp` ≤ `artifact_preview.expires_at`.
- **CT-Inbound-01:** Given a button-tap webhook payload from the WhatsApp adapter, the parser produces a Correction with `action_button` matching the tapped enum, gateway-internal `parse_method: "button"`, `parse_confidence: 1.0`.
- **CT-Inbound-02:** Given a free-text message matching a known synonym list, the parser produces `parse_method: text_match` and the correct `button_action`.
- **CT-Inbound-03:** Given an adversarial free-text message that is ambiguous, the parser falls through to LLM parse or emits a follow-up question (never silently drops).
- **CT-Inbound-04:** Given a duplicate `message_id` webhook, the second inbound is deduplicated; no second Correction is emitted (I-02).
- **CT-Idempotency-01:** Replaying the same `ReviewPacket` delivery (same `idempotency_key`) does not produce a second outbound message (I-05).
- **CT-Expiry-01:** A reply to a packet whose `expires_at` is in the past produces a `stale_reply` event and no Temporal signal (I-04).
- **CT-Signal-01:** A resolved Correction maps 1:1 to exactly one Temporal signal (`CaseApproved` or `CaseRejected`); the signal payload's `idempotency_key` matches `{case_run_id}:{decision_point_id}:{approved|rejected}:1`.

## 4. WhatsApp Connector

### 4.1 Approach: whatsmeow embedded in gateway

**Chosen candidate:** The `whatsmeow` Go library embedded directly in the gateway process, following the pattern cited in the product spec (the OpenClaw and Hermes WhatsApp connector references and the Tencent developer article). whatsmeow implements WhatsApp's multi-device WebSocket protocol, enabling per-tenant session management without requiring WhatsApp Business API verification. Because the gateway is Go, whatsmeow is used as a library — no REST wrapper or sidecar process is needed.

**Reasoning:**
- Multi-device protocol: stable across phone on/off cycles; no Business API application or verification friction at demo/pilot stage.
- Native Go library: no process boundary, no HTTP serialization overhead, no sidecar container per tenant.
- Per-tenant session: each tenant gets a dedicated whatsmeow session (QR-paired) managed as a goroutine within the gateway. This maps cleanly to the per-tenant number model.
- Used by the cited OpenClaw and Hermes WhatsApp connector paths for non-Business-API deployments.

**Flagged as reversible.** The gateway uses a `ChannelAdapter` interface (Section 4.5). Swapping WhatsApp Business API in later requires implementing the interface, not touching the rest of the system.

**Runner-up:** mautrix-whatsapp (Matrix bridge). More complex, adds a Matrix homeserver dependency; better for multi-protocol scenarios in Phase 2. Not chosen for Phase 1.

**Not chosen:** WhatsApp Business API. Deferred per the product spec — verification and approval friction makes it infeasible at demo/pilot stage.

### 4.2 Per-tenant number binding

Each tenant gets one dedicated WhatsApp number. The binding is owned by a gateway-side routing table in Postgres:

```sql
-- channel_bindings (gateway-owned)
tenant_id        TEXT NOT NULL,
channel          TEXT NOT NULL CHECK (channel IN ('whatsapp', 'telegram')),
provider_number  TEXT NOT NULL,   -- E.164 for WhatsApp; bot_token for Telegram
session_status   TEXT NOT NULL,   -- 'active', 'qr_needed', 'disconnected'
linked_phone     TEXT,            -- operator's phone number that receives the packets
created_at       TIMESTAMPTZ NOT NULL,
updated_at       TIMESTAMPTZ NOT NULL,
PRIMARY KEY (tenant_id, channel)
```

**Lifecycle ownership (RESOLVED-N4):** ctrl-plane provisions the WhatsApp session storage (volume + KEK) as part of `TenantProvisioningWorkflow`. The gateway operates the whatsmeow session in-process — no separate bridge container exists. ctrl-plane owns the tenant lifecycle (provision, suspend, deprovision); the gateway owns the session runtime (connect, reconnect, heartbeat, message I/O).

This addresses devils-advocate UE-CP-3 ("provisioning workflow lists 12 steps, none of which mention provisioning WhatsApp sessions").

### 4.3 Lifecycle handoff: ctrl-plane ↔ gateway (R3)

| Step | Owner | Action |
|---|---|---|
| 1 | ctrl-plane | `TenantProvisioningWorkflow` adds an `InitWhatsAppSession` activity: creates the `wa-session-data-{tenant_id}` volume, issues the session-key KEK, stores it in gateway-only secret scope |
| 2 | ctrl-plane → gateway | Calls `POST /channel-bindings` with `{tenant_id, channel: "whatsapp", operator_phone}` |
| 3 | gateway | Inserts the `channel_bindings` row, initializes a whatsmeow session goroutine for this tenant, returns the QR-pairing payload |
| 4 | ctrl-plane | Surfaces the QR-pairing payload to the admin/operator via the ctrl-plane-owned admin UI (RESOLVED-AdminUX-QR). For re-fetch on QR expiry, ctrl-plane admin UI calls the gateway's `GET /channel-bindings/:tenant_id/qr` endpoint (gateway-owned, mTLS-authenticated per ctrl-plane §6.2) which returns the current QR bytes if `session_status == 'qr_needed'` or 410 if already paired. |
| 5 | operator | Scans QR with their personal WhatsApp app |
| 6 | whatsmeow → gateway | whatsmeow reports session established; gateway flips `session_status = 'active'` |
| 7 | (operating) | Gateway operates the session in-process: whatsmeow handles reconnect natively; gateway monitors session health, manages alerting, and routes message I/O |
| 8 | ctrl-plane | On tenant deprovision: `DeprovisionWhatsAppSession` activity signals gateway to disconnect the session, destroys the `wa-session-data-{tenant_id}` volume; ctrl-plane calls gateway's `DELETE /channel-bindings/{tenant_id}/whatsapp` to clean up the binding row |

**Why ctrl-plane provisions storage but gateway runs the session:**
1. Symmetry with Hermes/Temporal-worker provisioning. ctrl-plane owns all per-tenant resource lifecycle (volumes, secrets, KEKs).
2. The gateway embeds whatsmeow directly — no separate container, no Docker-daemon access needed. The session runs as a goroutine within the gateway process.
3. Devils-advocate UE-CP-3 explicitly requested WhatsApp provisioning be part of `TenantProvisioningWorkflow`.

### 4.4 Session storage and encryption-at-rest

The gateway persists WhatsApp session keys (whatsmeow SQLite store) on a dedicated Docker volume named `wa-session-data-{tenant_id}`. The naming mirrors exec-plane's INV-H2 volume convention so volume-tenant binding is symmetric across the system.

**Encryption at rest (R4 reconciled with ctrl-plane §5.7 step 2):** the volume is encrypted using a per-tenant KMS-managed key, provisioned by ctrl-plane and stored in the **gateway secret scope** (`gateway/tenants/{tenant_id}/wa_session_kms_key`), not the tenant secret scope. R3 had this in the tenant scope; ctrl-plane §5.7 step 2 places it in the gateway scope, on the same rationale as the Temporal signal credential (§10.4 above): provisioning ownership rests with ctrl-plane, and ctrl-plane chooses the gateway scope to bound the read surface to the gateway component. Yielded.

**Volume lifecycle:** created at provisioning by ctrl-plane; never reused across tenants; destroyed by ctrl-plane on tenant deprovisioning. The gateway verifies the volume's `tenant_id` label matches the session's tenant at initialization — mismatch halts startup. This mirrors exec-plane INV-H4 and ctrl-plane INV-WA-03.

**Key rotation:** 90-day cadence managed by ctrl-plane's secret-rotation scheduler (ctrl-plane §5.7). Online: new key version added; gateway re-encrypts the session store on next idle window; old version retained for the rotation window. No operator re-pairing required.

### 4.5 ChannelAdapter interface (RESOLVED-C7 — narrowed)

**Devil's-advocate OE-01 review:** A generic plugin/channel-registry abstraction at N=2 channels is premature. Round 1 had `send` + `ack` + `getSessionStatus` on the interface, which forces incompatible semantics across WhatsApp (multi-device session) and Telegram (stateless bot). Round 2 narrows the interface to *only what is genuinely shared* — the test seam, not the future-channel abstraction.

```go
type ChannelAdapter interface {
	// SendOutbound renders a ReviewPacket and delivers it via the channel's transport.
	// Returns a DeliveryReceipt the gateway uses for outbound idempotency tracking.
	SendOutbound(ctx context.Context, msg OutboundMessage) (DeliveryReceipt, error)

	// NormalizeInboundWebhook normalizes a raw provider webhook payload into the
	// gateway's canonical InboundMessage. Pure function; no I/O.
	NormalizeInboundWebhook(rawPayload []byte) ([]InboundMessage, error)
}

type OutboundMessage struct {
	TenantID            string       `json:"tenant_id"`
	RecipientIdentifier string       `json:"recipient_identifier"` // E.164 phone for WhatsApp, chat_id for Telegram
	PacketID            string       `json:"packet_id"`
	BodyText            string       `json:"body_text"`            // rendered at send time per channel capability
	Buttons             []ButtonSpec `json:"buttons"`              // adapter selects rendering: native buttons | list | numbered text
	AttachmentURL       string       `json:"attachment_url,omitempty"`
	IdempotencyKey      string       `json:"idempotency_key"`
}

type InboundMessage struct {
	TenantID          string `json:"tenant_id"`           // populated by gateway via channel_bindings, NOT by adapter
	SenderIdentifier  string `json:"sender_identifier"`
	ProviderMessageID string `json:"provider_message_id"`
	Channel           string `json:"channel"`             // "whatsapp" | "telegram"
	ButtonPayload     string `json:"button_payload,omitempty"`
	FreeText          string `json:"free_text,omitempty"`
	ReceivedAt        string `json:"received_at"`         // ISO 8601
}
```

**What lives on concrete classes, not the interface:**
- WhatsApp: whatsmeow session lifecycle, QR pairing, reconnect heartbeat, button rendering capability detection (`interactive_buttons | list_message | text_fallback`), multi-device disconnect handling.
- Telegram: callback_query acknowledgment, inline keyboard rendering, bot token validation.

**Why the abstraction earns its weight:** the test suite `CT-Channel-01` runs the same `ReviewPacket → render → parse round-trip` contract suite against both backends. Without a shared interface, the contract would have to be hand-mirrored in two places and would drift. The two-method interface is the smallest surface that lets the test seam exist.

**Anti-scope (deliberately excluded):** no `ChannelRegistry`, no plugin discovery, no factory abstraction, no channel-config DSL. Phase 1 has exactly two adapters; if a third arrives, this section is revisited.

### 4.6 Button rendering strategy (RESOLVED-OE-OU-1 — pinned, not negotiated)

**Devils-advocate OE-OU-1:** Round 2 specified runtime capability negotiation across three rendering modes. whatsmeow's button-rendering capability is fixed per library version; runtime negotiation is dead code at Phase 1.

**R3 simplification.** Each adapter pins exactly one rendering mode per its current whatsmeow/library version:

| Channel | Pinned mode | Notes |
|---|---|---|
| WhatsApp (current whatsmeow version) | **List Message** | Tap-to-open picker with up to 10 rows; covers the 6-button set with one consistent UX |
| WhatsApp (numbered text fallback) | Active only if a future whatsmeow version regresses List Message support | Documented but unused in Phase 1 |
| Telegram | Inline keyboard | Native 6-button support |

If whatsmeow's capability changes (e.g., upgrading to a version with native interactive buttons), the rendering mode is updated as part of the gateway build — not at runtime via capability advertisement. No `ButtonCapability` enum, no negotiation layer.

When the numbered text fallback is active, the parser accepts numeric replies (`"1"` through `"6"`) as button-equivalent, with gateway-internal `parser_method: "button_fallback"` and `parser_confidence: 1.0`. The wire-level envelope does not distinguish these from native button taps (both populate `action_button` identically).

### 4.7 WhatsApp Session Lifecycle (RESOLVED-WA-SPOF)

This section is the explicit reconnect/buffering/alerting contract demanded by devils-advocate UE-01. The whatsmeow session is the system's most failure-prone link; it must be specified end to end.

#### 4.7.0 Invariants (R4 promoted to head of section)

| # | Invariant |
|---|---|
| WA-INV-1 | `channel_bindings.session_status` transitions only via the triggers listed in §4.7.1; any other transition is a bug. |
| WA-INV-2 | When `session_status != active`, no outbound message is sent via whatsmeow; packets are queued (§4.7.4). |
| WA-INV-3 | `outbound_queue` depth is bounded by `OUTBOUND_QUEUE_MAX = 100` per tenant; overflow tombstones oldest, never silently drops without audit. |
| WA-INV-4 | No inbound message and no outbound packet is dropped silently. Every drop emits a typed audit event (`packet_tombstoned`, `unbound_inbound_dropped`, `stale_reply`, `correction_dead_lettered`, `security_violation_inbound`) and an alert per §4.7.3 if severity warrants. |
| WA-INV-5 | Across any session disconnect of duration ≤ 30 minutes, no message is lost (I-11 from §3 invariants). |

| WA-INV-7 | `wa-session-data-{tenant_id}` volume is per-tenant; cross-tenant key reuse fails (ctrl-plane INV-WA-03). |

Each invariant has a contract test in §3 (CT-Outage-01 covers WA-INV-2/3/4/5; CT-TenantBinding-01 covers WA-INV-7's tenant-isolation aspect; the state-transition matrix below is enforced by a property test).

#### 4.7.1 Session states

```
qr_needed → connecting → active ⇆ disconnected → suspended (manual)
                            ↓
                       deprovisioned
```

Stored in `channel_bindings.session_status`. Transitions are the only legal state changes:

| From | To | Trigger |
|---|---|---|
| `qr_needed` | `connecting` | Operator scans QR; whatsmeow initiates handshake |
| `connecting` | `active` | whatsmeow reports session established (connection OK) |
| `active` | `disconnected` | whatsmeow connection lost; threshold exceeded (60s) |
| `disconnected` | `active` | whatsmeow reports session restored |
| `disconnected` | `suspended` | Disconnected > 24h, or rate-limit/abuse trigger |
| any | `deprovisioned` | Tenant deprovisioning |

#### 4.7.2 Reconnect contract

- whatsmeow handles low-level reconnect internally (websocket re-establishment, key re-handshake). The gateway does not re-implement this logic.
- The gateway monitors whatsmeow's connection status via its event handler callback. Three consecutive connection-lost events within 60s flips `session_status → disconnected`.
- On heartbeat restored, the gateway flips `session_status → active` and triggers outbound queue drain (§4.7.4).
- WhatsApp's multi-device protocol buffers inbound messages on the WhatsApp side for ~14 days. This is documented as a *property* of the protocol, not a *guarantee* of this system. The system's contract is at-most-30-minutes loss-free (I-11); longer outages depend on WhatsApp's server-side retention.

#### 4.7.3 Alerting hooks (RESOLVED-ALERT-SINK; R4 RESOLVED-EMIT-ALERT-SCHEMA)

**Alert types the gateway emits, mapped to ctrl-plane §9.6.1 `AlertPayload` (severity, category, code) verbatim:**

| `code` | `severity` | `category` | Triggered when |

| `wa_session_disconnect_critical` | `critical` | `session` | Disconnected for > 24 h; auto-transitioned to `suspended` |
| `wa_session_crash_loop` | `error` | `session` | whatsmeow session reports 5 failures in 10 min |
| `security_violation_inbound` | `critical` | `security` | Inbound webhook payload `tenant_id` disagrees with `channel_bindings.tenant_id` (§4.9) |

#### 4.7.4 Outbound queue (durable, per-tenant)

When `session_status != active`, outbound packets cannot be sent. Rather than failing the `SendApprovalPacket` activity, the gateway enqueues them in Postgres:

```sql

idempotency_key    TEXT NOT NULL,

rendered_payload   JSONB NOT NULL,   -- pre-rendered OutboundMessage

attempted_at       TIMESTAMPTZ,
delivered_at       TIMESTAMPTZ,
status             TEXT NOT NULL CHECK (status IN ('pending','delivering','delivered','tombstoned')),
attempt_count      INT NOT NULL DEFAULT 0,
UNIQUE (tenant_id, idempotency_key)
```

**Queue invariants:**
- Bounded depth: max 100 pending packets per tenant. When the limit is reached, the *oldest* pending packet is tombstoned with audit event `outbound_packet_tombstoned`. This keeps the failure visible rather than letting the queue grow unbounded.
- Drain order: FIFO by `enqueued_at` within a tenant.
- Drain trigger: `session_status` transitions to `active` OR a manual operator-ops resume.
- Drain rate: respects the per-`(tenant_id, operator_phone)` rate limit defined in §4.8.
- A tombstoned packet emits a `correction_expired`-equivalent audit event so the workflow's signal-wait-timeout path still terminates cleanly per exec-plane §3.5.

#### 4.7.5 Inbound during gateway outage

If the *gateway* is down: WhatsApp's multi-device protocol buffers inbound messages server-side. On gateway recovery, whatsmeow drains pending messages; redelivery dedup via `inbound_dedup` (§9.2) handles double delivery. No special handling needed.

#### 4.7.6 Contract test (CT-Outage-01)

Already listed in §3 contract tests. Test harness:
1. Bring up a gateway in Docker Compose; pair a test whatsmeow session.
2. Inject 5 `SendApprovalPacket` calls from a mock exec-plane.
3. Drop the whatsmeow WebSocket connection (simulate via firewall block) for 30 minutes.
4. During the outage, fire 3 webhook deliveries from a mock WhatsApp side.
5. Restore the connection.
6. Assert: 5 outbound packets delivered exactly once each (no duplicates, no losses); 3 `Correction` payloads emitted exactly once each; per-tenant queue empty; audit events for disconnect → reconnect present in order.

### 4.8 Rate limits and abuse handling

- Outbound rate limit: 1 message per `(tenant_id, operator_phone)` per 5 seconds; enforced in Redis.
- No automated retry of the same packet to the operator within 1 minute of a failed delivery.
- Bridge-level sending is serialized per session to avoid WhatsApp anti-spam detection.
- Inbound flood: if a tenant's operator sends >20 messages in 60 seconds with no matching `packet_id`, the gateway emits an abuse alert and dead-letters further messages until manual reset.
- Session abuse detection (unusual sending volume) triggers automatic `session_status = 'suspended'` and an internal alert.

### 4.9 Inbound webhook tenant-id validation (RESOLVED-Tenant-Validation)

This section answers devils-advocate INV-01 and SC-05 directly: the gateway must derive `tenant_id` from authenticated channel binding, never from message content.

**Validation flow on every inbound webhook:**

1. Webhook arrives. The whatsmeow event handler identifies the tenant via the `provider_number` that received the message.
2. Gateway looks up `channel_bindings` row by `provider_number`. The row's `tenant_id` is the **sole authoritative source**.
3. Any `tenant_id` field present in the webhook payload (e.g., a forged or replayed body) is **ignored** and not logged at the same level as the authoritative value.
4. If `provider_number` matches no `channel_bindings` row: return 200 (do not leak existence to the provider), drop the message, emit `unbound_inbound_dropped` audit event with the `provider_number`.
5. If `session_status != active` for the bound tenant: still process the inbound (idempotent dedup applies on reconnect), but flag with `processed_during_disconnect: true` in the audit event.

**Mismatch handling (devils-advocate SC-05 sub-scenario):** if the inbound webhook payload contains a `tenant_id` field that disagrees with `channel_bindings.tenant_id`, the gateway:
- Drops the message (no Correction emitted, no signal sent).
- Writes a `security_violation_inbound` audit event with both values, the `provider_number`, and the `provider_message_id`.
- Triggers an alert sink notification (high-severity).
- Does not 4xx the webhook (returns 200 to avoid signal-leakage to a potential attacker who controls the bridge).

This invariant (I-12) and CT-TenantBinding-01 enforce the rule.

### 4.10 Storage topology — gateway-owned tables (R4 RESOLVED-Storage-Topology)

The gateway maintains its own Postgres database (gateway-local; shared infra, not per-tenant). Tables in this DB are owned by the gateway component and are not part of any tenant's per-tenant DB nor the ctrl-plane DB. Inclusion in the cross-spec storage-topology table:

| Table | Database | Reader(s) | Writer(s) | TTL / lifecycle | Rationale |
|---|---|---|---|---|---|
| `channel_bindings` | Gateway-local Postgres | Gateway (every inbound message hot path; every outbound send) | Gateway (writes triggered by ctrl-plane `POST /channel-bindings` from `InitWhatsAppSession` activity) | Lifetime of tenant; row deleted on ctrl-plane `DELETE /channel-bindings/:tenant_id` at deprovisioning | Gateway is sole reader on the hot path — putting this in ctrl-plane DB would add a network hop on every inbound message. Not per-tenant — keying on `provider_number` requires a single multi-tenant table for the routing lookup (the bound number → tenant_id resolution is itself the multi-tenant operation). |
| `followup_session` | Gateway-local Postgres | Gateway parser (§7.2) | Gateway parser | TTL ≤ 1 hour; deleted on resolution, dead-letter, or expiry (I-10) | Ephemeral, gateway-only, never read by any peer. Per-tenant DB would be wrong (gateway has no per-tenant DB credentials, RESOLVED-AUDIT-EMIT). Ctrl-plane DB would be wrong (TTL'd ephemeral state pollutes the durable plane). |
| `outbound_queue` | Gateway-local Postgres | Gateway drain worker | Gateway packet handler | Bounded depth 100/tenant (WA-INV-3); rows deleted on delivery or tombstone | Same reasoning as `channel_bindings`. |
| `inbound_dedup` | Gateway-local Postgres | Gateway webhook handler | Gateway webhook handler | 24h TTL | Same reasoning. |
| `correction_signal_delivery` | Gateway-local Postgres | Gateway signal sender (dedup check) | Gateway signal sender | No TTL | Tracks gateway-side `signal_id` UNIQUE; matches exec §15 layer 1 dedup. |

**No PII flows from these tables to ctrl-plane DB.** `channel_bindings.linked_phone` is operator-bound PII; gateway hashes it for cross-system audit references, never sends raw phone to ctrl-plane (matches learning §13.5 PII-strip principles by analogy — gateway is upstream of cross-tenant aggregation).

**Cross-reference:** ctrl-plane §5.7 step 3 (`InitWhatsAppSession` calls gateway's `POST /channel-bindings`) is the binding write trigger. Ctrl-plane does not own this row; ctrl-plane is a caller.

## 5. Operator Authentication & Telegram Connector

### 5.0 Operator OTP/JWT (RESOLVED-OQ-06)

Per ctrl-plane §6.7, the **control-plane Auth Service is the sole issuer of operator JWTs**. The gateway is not an issuer.

**Gateway role in OTP:**

1. The Auth Service receives an OTP-initiate request (e.g., from an admin onboarding UI). It calls the gateway out-of-band: `POST /internal/messaging/deliver-otp` with `{tenant_id, phone_e164, otp, expires_at}`.
2. Gateway looks up the appropriate `channel_bindings` row, formats the OTP message, and sends via the appropriate channel (WhatsApp first; Telegram dev fallback).
3. Operator receives the OTP and submits it back through ctrl-plane's `POST /auth/operator/otp/verify` (the gateway is not in this path — verification is operator → ctrl-plane direct).
4. Auth Service issues the JWT per its §6.7 claims schema (`tid`, `sub`, `role: operator`, etc.).

**Why the gateway is the OTP-delivery channel but not the verifier:**
- The gateway already has the only out-of-band path to the operator's phone (the WhatsApp session). Reusing it for OTP delivery is the right division of labor.
- Verification is a credential operation; placing it on shared infra would put OTP secret material into the gateway. Ctrl-plane's Auth Service is the right owner per §6.7.

**TDD:**
- `CT-OTPDeliver-01`: Auth Service calls gateway's `POST /internal/messaging/deliver-otp`; assert WhatsApp message containing the OTP is sent to the bound `linked_phone`; gateway does not store the OTP value.
- See also `CT-OTPVerify→AuthService` in §3.1 for the verification side.

**Phase 1 scope.** Operators in Phase 1 do not have a browser-facing surface — they live in WhatsApp. The JWT is consumed by reviewer/admin tooling (ctrl-plane Rule Review Console) and any future operator web surface (out of scope for Phase 1). The OTP flow above is the wiring that lights up later UX.

### 5.1 Telegram Connector (dev/internal)

Telegram uses a standard Bot API webhook. A single bot token per tenant suffices for Phase 1 (one Telegram chat per tenant, used by internal developers and the tenant admin during setup).

The Telegram adapter implements the narrowed `ChannelAdapter` interface (§4.5):
- `sendOutbound` → Bot API `sendMessage` with `reply_markup` inline keyboard (6 buttons fit natively, no fallback needed).
- `normalizeInboundWebhook` → maps Bot API update payloads (text messages and `callback_query`) into `InboundMessage`.

Telegram-specific concerns kept off the shared interface (concrete-only): `answerCallbackQuery` for tap acknowledgment, `getMe` for health probing, bot token validation. These do not have analogues on the WhatsApp side and forcing them onto the interface would leak channel concerns.

Telegram bot provisioning is simpler than WhatsApp: no QR pairing, no whatsmeow session. A bot token is stored in the tenant's secret scope (Telegram is a tenant-scoped credential because the bot impersonates the tenant). The gateway stores a row in `channel_bindings` with `channel = 'telegram'`.

The Telegram channel is never the primary customer-facing channel. It shares the same packet schema, correction shape, and signal mapping as WhatsApp.

## 6. Outbound Review Packet — Detail

See the canonical JSON sketch in Section 2.1. Elaborations:

**`trigger.summary`:** A human-readable string generated by the execution plane and passed through the gateway unchanged. The gateway does not generate or modify this string.

**`facts`:** Ordered list. `confidence` is execution-plane-generated; gateway passes it through for potential rendering (e.g., low-confidence facts could be visually distinguished). Schema is fixed; rendering is up to the channel adapter.

**`planned_action.is_destructive`:** When `true`, the gateway MAY suppress autopilot UI and render a more prominent approval prompt. The execution plane sets this; the gateway enforces the UI constraint.

**`artifact_preview`:** If `inline_thumbnail_url` is present (e.g., a small image), the gateway sends it as a media attachment before the text packet. If only `preview_url`, the URL is embedded in the message text. `preview_url` is a signed JWT URL (see Section 8).

**`button_set`:** The enum values are a subset of the canonical six: `{approve, wrong_facts, wrong_action, missing_condition, use_different_template, add_note}`. The execution plane may omit buttons that don't apply to a given decision type (e.g., `use_different_template` is omitted for non-template decisions).

**`expires_at`:** The absolute time after which operator replies are stale. Set by the execution plane. The gateway stores this and enforces it on inbound.

**Message length limit:** WhatsApp limits messages to 4096 characters. If the rendered packet exceeds this, the gateway truncates the facts list (keeping the first N facts and appending a count of omitted facts) and trims the `planned_action.description`. It does not truncate the `preview_url` or button set. This truncation is logged as an audit event.

## 7. Inbound Reply Parser

### 7.0 Invariants (R4 promoted to head of section)

| # | Invariant |
|---|---|
| PARSER-INV-1 | The cascade is deterministic: identical (inbound message + open follow-up state) always produces identical (`action_button`, `parser_method`, `parser_confidence`, `parse_status`). No randomness, no time-of-day branching. |
| PARSER-INV-2 | An inbound message produces exactly one of `{button_tap_resolved, text_match_resolved, follow_up_emitted, dead_lettered}`. No silent drops. (Restates I-01.) |
| PARSER-INV-3 | The gateway never populates `parsed_conditions` or `parsed_action` (Stage B is learning's, RESOLVED-Parser-split). I-14. |
| PARSER-INV-4 | Phase 1 has no Stage 3 LLM parse (RESOLVED-OE-OU-2). The cascade has exactly four stages: button → text-match → follow-up → dead-letter. |
| PARSER-INV-5 | Multi-packet free-text correlation prefers a "which packet?" follow-up over a heuristic match (RESOLVED-UE-OU-2). Heuristic correlation is forbidden. |
| PARSER-INV-6 | Configurable knobs (`threshold_text_match`, follow-up TTL) have deterministic compiled-in defaults; an unset knob does not change behavior between runs. (CT-Threshold-Default-01.) |

### 7.1 Parse cascade (RESOLVED-Parser-split — gateway scope only)

**Parsing-pipeline ownership split:**

| Step | Owner | Notes |
|---|---|---|
| Classify inbound to `action_button` enum | **Gateway** | Button tap (Stage 1) and normalized text match (Stage 2) only in Phase 1 |
| Determine `scope_hint` from follow-up answer | **Gateway** | Maps "always" / "always when X" → `tenant`; "this case" → `case` |
| Detect ambiguity, emit follow-up questions | **Gateway** | Multi-turn flow held in gateway-local session |
| Dead-letter routing | **Gateway** | Hand-off to internal review queue |
| Semantic extraction → `parsed_conditions[]` and `parsed_action` | **Learning loop** (§3, §4 of `03-correction-loop.md`) | Requires access to `decision_point.agent_input`, `workflow_templates.decision_types` |
| Confidence scoring of the parse | **Learning loop** | Their `corrections.parse_confidence` column |
| Candidate matching, condition canonicalization | **Learning loop** | §4.2 of correction-loop spec |

The gateway does NOT populate `parsed_conditions` or `parsed_action`. Those columns are filled by learning's Stage B downstream.

**Defense:** the gateway lacks the workflow-condition vocabulary required for semantic parsing. That schema lives on the learning side.

**Gateway-side stages** (each stops at the first that produces a `parse_status: resolved`):

```
Stage 1: Button tap (interactive button or numbered fallback)
  - Inbound carries a buttonPayload OR matches "1".."6"
  - → parser_method: "button" or "button_fallback", parser_confidence: 1.0
  - action_button assigned directly

Stage 2: Exact/normalized text match
  - Free text normalized (lowercase, trim, strip punctuation)
  - Matched against locale-scoped synonym table per action_button enum
  - → parser_method: "text_match", parser_confidence: configurable knob (default ~0.95)

Stage 3: Follow-up question emitted
  - Stages 1–2 miss
  - Gateway emits a clarification (e.g., "Always wrong, or only this case?")
  - Multi-turn state held in follow-up session (§7.2)
  - Operator's answer re-enters Stage 1–2 with combined context
  - One follow-up round only; second-round ambiguity → Stage 4

Stage 4: Dead-letter
  - parser_method: null, parser_confidence: 0.0
  - parse_status: "dead_lettered" (gateway-internal)
  - Audit-logged; routed to internal review queue
  - No envelope emitted; no signal sent
```

**RESOLVED-OE-OU-2.** Devils-advocate flagged Stage 3 LLM parse as premature: the button set is narrow and structured by design, and button tap (Stage 1) covers the dominant case. **Phase 1 ships without LLM-assisted classification.** This removes the LLM-credentials surface from the gateway, removes a parse-pipeline cost/latency/observability layer, and tightens the failure surface.

**Phase 2 trigger.** If post-launch metrics show that text-match (Stage 2) misses on more than `text_match_miss_threshold` (configurable knob; not set in this spec) of free-text replies, LLM-assisted classification is reintroduced as a new Stage between current 2 and 3. The wire envelope already accommodates this — `parser_method: "llm"` is reserved.

**Confidence thresholds** (`threshold_text_match` for Stage 2 promotion to resolved) are runtime configuration knobs, not hardcoded values. **CT-Threshold-Default-01:** an unset knob produces a deterministic default (the spec value above) — required to address devils-advocate audit point on threshold contract tests.

### 7.2 Follow-up question flow

The follow-up question flow captures scope and condition information after a non-Approve correction. The two canonical follow-up questions are:
- **Scope qualifier:** "Always wrong, or only this case?" (key: `scope_qualifier`)
- **Signal qualifier:** "What signal should have changed the decision?" (key: `signal_qualifier`)

Which follow-ups to ask is determined by the `button_action`:
- `wrong_action` → ask `scope_qualifier`, then optionally `signal_qualifier`
- `missing_condition` → ask `signal_qualifier`
- `wrong_facts` → no follow-up (fact corrections are self-describing)
- `use_different_template` → no follow-up (template ID or name captured in free text)
- `add_note` → no follow-up

**Multi-turn state schema (gateway-owned, Postgres, TTL-bounded):**

```json
{
  "followup_session_id": "fus_abc",
  "tenant_id": "t_123",
  "packet_id": "pkt_7a3c",
  "decision_point_id": "dp_789",
  "pending_questions": ["scope_qualifier"],
  "answers_so_far": [
    { "question_key": "scope_qualifier", "answer_text": "always when client is new and photos are incomplete" }
  ],
  "partial_correction": { "action_button": "wrong_action", "free_text": "..." },
  "created_at": "2026-04-27T08:14:00Z",
  "expires_at": "2026-04-27T09:14:00Z"
}
```

**Flattening to learning's `follow_up_answer` (single string):** Learning §3 specifies `follow_up_answer` as a single string. When the gateway emits the Correction:
- If `answers_so_far` contains exactly one answer: use the `answer_text` verbatim.
- If multiple: concatenate them with newlines, in question order: `"<question_key_1>: <answer_1>\n<question_key_2>: <answer_2>"`.

This preserves the operator's words for learning's downstream semantic parser without inventing a structure-vs-string conflict. **RESOLVED-followup-shape:** single string per learning §3; multi-turn answers concatenated newline-separated. Resolved in Round 3.

TTL on follow-up sessions is 1 hour (configurable). On expiry:
- If at least `action_button` is resolved, the gateway emits the partial Correction with whatever `follow_up_answer` text has been collected, `scope_hint = null` if scope question unanswered, and `parse_status: resolved` (gateway-internal).
- If no `action_button` is resolved, the session is dead-lettered.

**Session keying:** `(tenant_id, packet_id)`. The gateway maintains at most one active follow-up session per `(tenant_id, packet_id)` pair. If the operator sends a new message before the follow-up is answered, the gateway appends context to the existing session (not a new session).

**Abandonment:** If the operator stops responding before completing the follow-up, the TTL expiry path applies (above).

### 8.1 What gets inline vs. linked

| Artifact type | Rendering approach |
|---|---|
| PDF (proposal, invoice) | Signed URL link; gateway embeds URL in message text |
| Draft email text (short, ≤300 chars) | Inline in message body |

### 8.2 Signed URL scheme

Preview URLs are JWTs signed with a per-tenant signing key. Claims:

```json
{
  "sub": "artifact_preview",
  "tenant_id": "t_123",
  "packet_id": "pkt_7a3c",
  "artifact_id": "art_def",
  "exp": 1745740800,
  "iat": 1745697600
}
```

The preview server validates:
1. JWT signature against the tenant's signing key.
2. `tenant_id` in claims matches the tenant derived from the serving path.
3. `exp` has not passed.

**Signing authority (RESOLVED-Preview-Signing).** The gateway is the sole signer of preview JWTs. The per-tenant signing key (Ed25519 keypair) is provisioned by ctrl-plane during `TenantProvisioningWorkflow` and stored in the tenant secret scope at `tenant/{tenant_id}/messaging/preview_signing_key`. The gateway IAM has read access to the `messaging/*` subtree (same path family as the Temporal signal credential, §10.4).

The execution plane generates `artifact_id` and stores the artifact in the tenant's object-store prefix; the gateway assembles the preview URL with the JWT. If a future use case requires exec-plane-initiated preview-link generation (e.g., reviewer-initiated re-review), the same signing key is read by exec-plane via the same secret-scope path; no shared signing service is introduced unless usage demands it.

**Key rotation.** Ctrl-plane rotates the signing keys on its standard rotation schedule. The JWT carries a `kid` claim; preview servers maintain the current and previous public key, so in-flight URLs signed by the previous key remain valid until their `exp`.

### 8.3 Mobile-first constraints

- Preview pages must be responsive and load under poor mobile network conditions.
- PDF preview: server-side render the first page to a JPEG thumbnail (stored in object store) at packet-generation time; embed as `inline_thumbnail_url`. The full PDF is accessible via the signed link.
- No JavaScript required for PDF preview pages — use a static HTML+`<img>` or PDF.js embed served from a CDN.
- Preview pages do not require operator login; the signed JWT in the URL is the authorization token.

## 9. Idempotency and Ordering

### 9.0 Invariants (R4 promoted to head of section)

| # | Invariant |
|---|---|
| IDEMP-INV-1 | Outbound delivery for `(tenant_id, packet_id)` is at-least-once with provider-side dedup (I-05). |
| IDEMP-INV-2 | Duplicate inbound webhook with the same `provider_message_id` is a no-op within the 24h dedup window; no second envelope is emitted (I-02). |
| IDEMP-INV-3 | A reply received after `packet.expires_at` produces a `stale_reply` audit event; no Temporal signal (I-04). |
| IDEMP-INV-4 | Out-of-order replies are correlated explicitly (button payload carries packet_id; free-text uses follow-up session); ambiguous correlation triggers a "which packet?" follow-up (PARSER-INV-5). |
| IDEMP-INV-5 | Webhook controller returns 200 before processing; processing is async. Failure to return 200 promptly causes provider redelivery handled by IDEMP-INV-2. |
| IDEMP-INV-6 | Audit and alert emit calls carry idempotency keys per §9.6 row 5/6; duplicate calls are absorbed by ctrl-plane UNIQUE constraints, not retried by the gateway. |

### 9.1 Outbound dedup

Every outbound `ReviewPacket` has an `idempotency_key` structured as `{packet_id}:deliver:{attempt_number}`. Before sending, the gateway checks a Redis idempotency store:
- If the key is present and status is `delivered`: return immediately, no-op.
- If the key is present and status is `in_flight`: wait for result or timeout.
- If absent: insert with TTL (default: 24 hours), send, update status.

### 9.2 Inbound dedup

Every inbound webhook carries a provider `message_id` (WhatsApp message ID, Telegram `update_id`). The gateway maintains a dedup table:

```sql
-- inbound_dedup (gateway-owned)
provider_message_id  TEXT PRIMARY KEY,
tenant_id            TEXT NOT NULL,
received_at          TIMESTAMPTZ NOT NULL,
status               TEXT NOT NULL  -- 'processing', 'processed', 'dead_lettered'
```

Dedup window: 24 hours (TTL on the row). Duplicate delivery within the window is a no-op; the gateway returns 200 to the provider webhook.

### 9.3 Out-of-order replies and free-text correlation (RESOLVED-UE-OU-2)

WhatsApp delivers messages in order within a session; reordering is rare but possible on session re-establishment.

**Button callbacks** carry a `packet_id` in their payload (or, for the numbered text fallback, the recent `pending_button_render` row records which packet's buttons were just sent) — so correlation is exact.

**Free-text replies** are correlated as follows:

1. If exactly one active follow-up session exists for `(tenant_id, operator_phone)` → correlate to that packet.
2. If exactly one unresolved packet exists for `(tenant_id, operator_phone)` → correlate to that packet.
3. If multiple unresolved packets exist (devils-advocate UE-OU-2 case) → emit a "which packet?" follow-up message listing the packets by their trigger summaries; await operator's selection. State held in `pending_correlation` row keyed on `(tenant_id, operator_phone)` with TTL 5 min.
4. If still ambiguous after the operator's reply (e.g., reply is "the first one" but the listed order has rotated, or the reply is itself ambiguous text), the message is **dead-lettered** with audit event `inbound_correlation_failed`. No envelope emitted.
5. If no open session and no unresolved packet → dead-lettered with `unmatched_inbound_dropped`.

**Heuristic-fallback removal:** R2's "fallback to the most recently sent unresolved packet" is removed. The fallback could deliver an operator's reply to the wrong case if the operator was actually responding to an older packet — that's a correctness-affecting silent corruption. Asking is safer.

**CT-Correlation-01** asserts the "which packet?" follow-up + dead-letter on second-round ambiguity (see §3 contract tests).

### 9.4 Stale replies

If a reply arrives after `packet.expires_at`:
- The gateway writes a `stale_reply` audit event (with the full inbound message).
- The gateway sends the operator a message indicating the decision window has closed.
- No Temporal signal is emitted.

### 9.5 WebhookRetry handling

WhatsApp may retry message delivery if the gateway's whatsmeow connection is interrupted. The gateway processes inbound messages asynchronously via whatsmeow's event handler — the handler dispatches to a goroutine and returns immediately to avoid blocking the whatsmeow event loop.

### 9.6 Idempotency-key inventory (RESOLVED-N6)

The gateway uses four distinct idempotency keys, each with a defined dedup target. Where ctrl-plane's R3 composition rule applies, this section conforms.

| # | Key | Format | Stored on | Dedups against | Notes |
|---|---|---|---|---|---|
| 1 | Outbound packet delivery | `{packet_id}:deliver:{attempt}` | Redis (TTL 24h) | Duplicate outbound delivery for same packet | `attempt` increments on retry; first delivery is always `:1` |
| 2 | Inbound webhook | `{provider_message_id}` | `inbound_dedup` table (TTL 24h) | Duplicate webhook redelivery from the bridge or provider | Returns 200 on duplicate; no envelope emitted |
| 3 | Envelope `idempotency_key` (= corrections row UNIQUE) | `sha256(tenant_id \| decision_point_id \| packet_id \| action_button)` per ctrl-plane §17.2 | learning's `corrections.idempotency_key` (UNIQUE) | Duplicate `corrections` row from PersistCorrection | Matches learning §3.7 invariant 3; matches §2.2 |
| 4 | Temporal signal `signal_id` | `sha256(tenant_id \| packet_id \| action_button \| raw_inbound_message_id)` per ctrl-plane §17.2 | exec-plane's signal-handler dedup state | Duplicate signal delivery from gateway retries or Temporal redelivery | Matches devils-advocate INV-06; matches §2.2 |
| 5 | Audit-emit calls | `audit:{tenant_id}:{event_type}:{occurred_at_epoch}:{ref_id}` | ctrl-plane `audit_events.idempotency_key` (UNIQUE) | Duplicate audit-event ingest | Per learning §11.2 |

### 9.7 Audit-event emission (RESOLVED-AUDIT-EMIT)

**The gateway calls ctrl-plane's `POST /internal/audit/events`. The gateway never holds per-tenant DB credentials.**

**Asymmetry with exec-plane (intentional and defended).** Per ctrl-plane §9.6, exec-plane writes audit events directly to its tenant's `audit_events` table because exec-plane is per-tenant infrastructure with naturally per-tenant DB credentials. The gateway is shared infrastructure: a single process handling all tenants' inbound and outbound. Giving the gateway direct DB write access to every tenant's audit table would put per-tenant DB credentials inside a shared process — a blast-radius regression. Going through ctrl-plane's `emit-audit` API lets the gateway hold a single ctrl-plane-side credential and lets ctrl-plane apply per-call tenant scope checks before INSERT.

## 10. Approval Signal Contract with Execution Plane

### 10.0 Invariants (R4 promoted to head of section)

| # | Invariant |
|---|---|
| SIGNAL-INV-1 | Gateway calls Temporal directly via the SDK; no HTTP routing through ctrl-plane (RESOLVED-C3, ratified). |
| SIGNAL-INV-2 | Exactly one Temporal signal per resolved envelope (I-15). |
| SIGNAL-INV-3 | Signal payload is the full 16-field envelope (§2.2); `PersistCorrection` builds the corrections row from it without any callback to the gateway (exec INV-A5; CT `test_persist_correction_from_signal_alone`). |
| SIGNAL-INV-4 | Two-layer dedup: gateway `correction_signal_delivery.idempotency_key` UNIQUE plus exec-plane signal-handler `signal_id` dedup. Either alone is insufficient (devils-advocate INV-06). |
| SIGNAL-INV-5 | `expires_at` is one value, two enforcement points: exec-plane authoritative for workflow state transition; gateway advisory for reminders and stale-reply detection (RESOLVED-Expiry-Sweep). |
| SIGNAL-INV-6 | Per-tenant signal-only Temporal credential, stored in gateway secret scope per ctrl-plane §6.8; cannot start, terminate, reset, or describe-workflow (I-19). |

See §2.2 for the canonical `ApprovalSignalEnvelope` (RESOLVED-C13) and §2.3 for the direct-to-Temporal rationale (RESOLVED-C3). This section covers delivery flow, timeout, escalation, and the gateway-held Temporal credential model (RESOLVED-OQ-NEW-5).

### 10.1 Delivery flow (R3 — single envelope, no separate Correction post)

1. Gateway constructs the `ApprovalSignalEnvelope` (§2.2) with all 16 fields populated.
2. Gateway validates the envelope schema and that the gateway-internal `parse_status` is `resolved`.
3. Gateway selects the per-tenant Temporal credential (§10.4) keyed by envelope's `tenant_id`.
4. Gateway calls `SignalWorkflow(workflow_id=case_run_id, signal_name=CaseApproved|CaseRejected, payload=envelope)` on the tenant's task queue.
5. Gateway records `signal_delivered_at` in `correction_signal_delivery(tenant_id, idempotency_key UNIQUE, signal_id, signal_delivered_at)`.
6. Gateway updates `inbound_dedup` row to `processed`.
7. Gateway emits `correction_received_at_gateway` audit event via ctrl-plane's `POST /internal/audit/events` (§9.6).

There is no separate POST to learning. Exec-plane's `PersistCorrection` activity is the sole writer of the `corrections` row (RESOLVED-C10), reading the envelope payload directly from the signal.

**Exactly-once delivery:** two-layer dedup (devils-advocate INV-06):
- Layer 1 (gateway): `correction_signal_delivery.idempotency_key` UNIQUE prevents duplicate signal emission.
- Layer 2 (workflow): exec-plane signal handler dedups on `signal_id` (INV-A2).

### 10.2 Timeout / "operator never replied" (RESOLVED-Expiry-Sweep)

`expires_at` is **a single value** set by exec-plane on the ReviewPacket. It corresponds 1:1 to exec-plane's `signal_wait_timeout` for the workflow type (12–48h per workflow, exec-plane §3.5). It is not two timers — it is one deadline enforced from two sides:

- **Exec-plane side (authoritative, workflow-state-transition):** Temporal `signal_wait_timeout` fires; workflow transitions to `AwaitingCorrectionTimeout` per exec-plane §3.5; the workflow path is decided by exec-plane logic.
- **Gateway side (advisory, operator UX):** gateway runs a sweep on `expires_at` for the purposes of (a) sending a reminder to the operator before expiry (per §10.3), (b) marking inbound replies arriving after expiry as `stale_reply` and not signaling them. The gateway does **not** signal Temporal on expiry; Temporal's own timer handles state transition.

If a reply arrives between gateway-detected expiry and Temporal-detected timeout (the two are not perfectly synchronized), the gateway treats it as `stale_reply`. Temporal will eventually time out and the workflow path proceeds without a signal. The operator receives a "decision window has closed" message.

### 10.3 Escalation

If the operator has not replied within `(expires_at - now) < escalation_threshold` (configurable), the gateway sends a reminder message. At most one reminder per packet.

### 10.4 Gateway Temporal credential consumption (RESOLVED-OQ-NEW-5; R4 reconciled with ctrl-plane §6.8)

**Invariants (R4 promoted to head of section):**
- `I-16` (R3): Per-tenant credential, loaded keyed by inbound `tenant_id`.
- `I-19` (R3): Credential cannot start, terminate, reset, or describe-workflow; only signal.
- `GW_CRED_01..04` (ctrl-plane §6.8): Cross-tenant credential reuse must fail; rotated cred becomes unusable within 1h grace.

**Storage location (R4 yielded to ctrl-plane §6.8).** Per-tenant signal-only credentials are stored in the **gateway-only secret scope** (provisioned in ctrl-plane §5.2 step 3), **not** the per-tenant secret scope used by the execution plane. This is ctrl-plane's R3 §6.8 decision (storage row: "Gateway-only secret scope ... NOT the tenant secret scope used by the execution plane"). My R3 reasoning (symmetry with exec-plane's tenant-scope pattern) is reversed: ctrl-plane provisions, ctrl-plane chooses where the gateway reads from. The R3 path `tenant/{tenant_id}/messaging/temporal_signal_client` is replaced with `gateway/tenants/{tenant_id}/temporal_signal_client`.

**Provisioning.** Per-tenant Temporal credentials are provisioned by **ctrl-plane's `TenantProvisioningWorkflow`** (ctrl-plane §6.8) and rotated on a 24-hour cadence by ctrl-plane's secret-rotation scheduler. Gateway is a consumer only.

**Credential capability scope** (ctrl-plane §6.8 GW_CRED rows; reproduced for traceability):
- Permits: `SignalWorkflow` only, on task queues matching `victoria.tenant.{this_tenant_id}.*`.
- Forbids: `StartWorkflow`, `TerminateWorkflow`, `CancelWorkflow`, `SignalWithStartWorkflow`, `ResetWorkflow`, `DescribeWorkflowExecution` (admin), `QueryWorkflow`, namespace admin, history queries.
- Forbids: any operation against another tenant's task queues.

**Loading model (per-signal):**

```
on inbound resolved (envelope ready to send):
  tenant_id = envelope.tenant_id  // from channel_bindings, NEVER from payload (I-12)
  cred = secret_cache.get(tenant_id, ttl=15min)
         or load_from_secret_store("gateway/tenants/{tenant_id}/temporal_signal_client")
  client = TemporalClient(cred, target_task_queue_prefix="victoria.tenant.{tenant_id}.")
  client.SignalWorkflow(...)
```

Cred cache is in-memory, TTL 15 min, keyed by `tenant_id`. Rotation flow per ctrl-plane §6.8: gateway watches its secret scope (or polls every 5 minutes); on detecting a new version, sets it as active for new connections; old cred remains valid in the 1-hour grace window.

**Contract test CT-TemporalCred-01:** load tenant A's cred; attempt `StartWorkflow`, `TerminateWorkflow`, `ResetWorkflow`, `DescribeWorkflow`, and `SignalWorkflow` against tenant B's queue. Assert all five fail with permission-denied. Then `SignalWorkflow` against tenant A's own queue succeeds. (I-19, ctrl-plane GW_CRED_01..03.)

**OPEN-Temporal-IAM (held open).** Ctrl-plane §6.8 specifies the credential model but does not certify that Temporal's IAM natively supports "SignalWorkflow on task-queue glob, nothing else." Gating question: does the deployed Temporal version (OSS vs. Cloud) support per-namespace per-action ACLs at the granularity of `SignalWorkflow` only? If not, the fallback is the signal-proxy sidecar described in operator-ux R3 §10.4 (held over from R3) with the same observable contract; the gateway code path is unchanged.

## 11. Failure Modes

| Failure | Detection | Response |
|---|---|---|
| WhatsApp session disconnect | whatsmeow connection lost; `session_status → disconnected` (per §4.7) | Alert internal ops at 5/60/1440-min thresholds; queue outbound in `outbound_queue` (bounded depth 100/tenant); drain on reconnect (§4.7.4) |
| whatsmeow session failure | Repeated session errors within the gateway process | Gateway reconnects automatically; whatsmeow store on `wa-session-data-{tenant_id}` volume preserves session keys; if 5 failures in 10 min, auto-suspend |
| Outbound queue full | `outbound_queue` depth = 100 | Tombstone oldest pending packet; emit `outbound_packet_tombstoned` audit event; the corresponding workflow's signal-wait-timeout will eventually expire and Temporal handles workflow-state transition |
| Inbound webhook tenant mismatch | Payload `tenant_id` differs from `channel_bindings.tenant_id` | Drop message; emit `security_violation_inbound` audit event; alert; return 200 to provider (no signal leak) |
| Message too long | Gateway pre-checks rendered length before send | Truncate facts list; log truncation event; send with truncation notice |
| Attachment upload failure | whatsmeow returns error on media upload | Send text-only packet with inline URL fallback; log failure |
| Rate limit hit (WhatsApp) | whatsmeow or provider returns rate-limit error | Exponential backoff; emit `rate_limit_hit` metric; do not retry indefinitely |
| Message ordering corruption | Gap in sequence numbers detected | Log sequence anomaly; continue processing with best-effort ordering; alert if gap > threshold |
| Follow-up session collision | Two replies for same `(tenant_id, packet_id)` before session resolved | Append to existing session; do not create duplicate |
| Multi-packet correlation ambiguity | Free-text reply with multiple unresolved packets (R3 RESOLVED-UE-OU-2) | Emit "which packet?" follow-up; dead-letter on still-ambiguous second-round reply |
| Signed URL expiry mismatch | Preview page receives expired JWT | Return 410 Gone; do not redirect to login |
| Ctrl-plane audit/alert API down | gateway `POST /internal/audit/events` or `/alerts/emit` returns 5xx | Buffer locally in Postgres `outgoing_audit_buffer` / `outgoing_alert_buffer`; retry with exponential backoff; never lose; never write to per-tenant DB as a fallback |
| Temporal SignalWorkflow non-retryable error | e.g., workflow already completed before signal | Emit `signal_delivery_failed` audit event; notify operator that the case is closed |

## 12. Privacy and Safety

- **PII in messages:** Operator-facing packets contain operator's own business data. The gateway does not redact content in transit. PII at rest is subject to tenant-scoped encryption keys.
- **Audit log redaction:** The audit log stores `raw_inbound_message_id` and the emitted Correction payload. Raw free-text from the operator is stored in learning's `corrections.free_text` and `corrections.follow_up_answer` columns. Retention and redaction policy is set per tenant by ctrl-plane §9.5.
- **What is logged vs. redacted:** Message body text is logged in the corrections record by exec-plane's PersistCorrection. WhatsApp session keys are stored on the `wa-session-data-{tenant_id}` volume, not in the application database. Phase 1 ships without LLM-assisted parse, so there are no LLM-parse-input residuals to retain.
- **Artifact preview access:** Preview URLs are tenant-scoped JWTs. No preview page returns data from another tenant. Server-side validation enforces this on every request.
- **No PII in button payloads:** Button enum values carry no operator data. Free text is in a separate field.

### 16.4 Pre-commit invariants (CI gates)

The following must hold on every commit; CI fails otherwise.

- No inbound webhook handler accepts a payload-supplied `tenant_id`; `tenant_id` is derived solely from `channel_bindings` lookup (I-12).
- No code path writes directly to a tenant's `audit_events` table; all audit emission goes through `POST /internal/audit/events` (I-17, RESOLVED-AUDIT-EMIT).
- `ApprovalSignalEnvelope` JSON validates against the 16-field JSONSchema; no field added or removed without spec update (I-13).
- `parse_status: needs_followup` and `parse_status: dead_lettered` envelopes never reach the Temporal signal emitter (I-07).
- Gateway holds no per-tenant database credentials (I-17).
- Default-deny on missing/unknown ReviewPacket `mode` field (I-18).
- Per-tenant Temporal signal credential cannot start, terminate, reset, or describe-workflow (I-19).
- Phase 1 has no LLM-assisted parser (PARSER-INV-4); CI rejects any code path importing an LLM client into the parser module.
